# Customer Data Boundary Policy Note

## 1. Назначение

Документ фиксирует проектное допущение о границе customer operational data, tenant/project data, anonymized/generalized records и aggregated signals. Эта предварительная policy note требует юридической проверки и не применяется без договорного основания.

## 2. Базовый принцип

Данные Заказчика по умолчанию остаются в локальном контуре Заказчика. Передача данных за пределы локального контура отключена по умолчанию и не возникает автоматически из факта поставки кода, модульной архитектуры, дорожной карты, БФТ, ТЗ или протокола испытаний.

## 3. Уровни данных

1. Customer operational data — исходные рабочие данные Заказчика остаются в customer contour и не покидают локальную среду без отдельного правового основания.
2. Pseudonymized tenant data — псевдонимизированные записи допустимы только внутри tenant/project boundary и не считаются безопасными для внешней передачи сами по себе.
3. Anonymized/generalized data — очищенные и обобщенные события, из которых удалены запрещенные поля и снижена детализация чувствительных атрибутов.
4. Aggregated signals — агрегированные показатели, допустимые только после отдельного согласия, агрегирования и re-identification risk check.

## 4. Запрет автоматической передачи

Наличие локального продукта, validators, scripts, dataset schemas или AI-related roadmap не означает автоматический сбор, выгрузку, передачу или повторное использование данных Заказчика. Любой выход за пределы локального контура требует отдельного согласия, договора или иного применимого правового основания.

## 5. Практическое следствие

До юридической проверки проект должен проектироваться как local-first: telemetry, dataset contribution, external upload, AI training и commercial data use считаются отключенными по умолчанию.
