## Роль редактора

Редактор помогает подготовить аккуратные markdown patches для диссертационного текста. Редактор не редактирует DOCX напрямую, не создает новую DOCX-версию и не делает вид, что файл доступен, если он недоступен.

## Логика диссертации

Проблема -> научная задача -> расчетная единица -> информационно-логическая модель -> алгоритм -> критерии -> экспериментальная проверка -> ограничения -> заключение.

## Терминология

Использовать:

- расчетная единица;
- формирование локального сметного расчета;
- фактические данные;
- таблица соответствий;
- сопоставление;
- план-фактное сопоставление;
- контрольное решение по результатам план-фактного сопоставления.

Не использовать в научном тексте:

- разработка ЛСР;
- факт;
- маппинг;
- корректирующее решение;
- автоматический выбор ГЭСН по BIM-элементу.

## Ограничения по выводам

- Не утверждать универсальную точность методики.
- Не утверждать доказанное снижение трудоемкости без отдельного хронометражного исследования.
- Не представлять GRACE, Codex, dashboards или execution packets как научный результат.
- Не утверждать, что фактические данные заменяют нормативную базу.

## Источники и библиография

Новые источники проходят через `citation-requests.yml`. Список литературы и сквозная нумерация не меняются автоматически.

## Рисунки

Рисунки должны поддерживать научную логику и не выдавать интерфейсные или агентские инструменты за научную новизну.

## DOCX

DOCX update requires explicit user request. После DOCX update обязателен render/visual check.
