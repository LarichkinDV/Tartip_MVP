# Политика учета источников

## 1. Принцип

`No source — no rule`.

Без проверяемого источника нельзя создавать active matching rule, подтверждать официальный КСИ-код, ГЭСН-код, ФСНБ-версию, нормативную единицу измерения, состав работ или resource composition.

Контур учета источников не создает нормативные правила и не подтверждает соответствие КСИ/ГЭСН/ФСНБ. Он только фиксирует поступление источника, его происхождение, версию, checksum, статус доверия и необходимость проверки.

## 2. Допустимые типы evidence

Official evidence может подтверждать официальные classifier/normative fields только после регистрации источника, checksum и пользовательской проверки.

Project dictionary может описывать внутренние work types, grouping rules, WorkPackage naming и project-only decisions. Он не подтверждает официальные нормативные поля.

User decision может фиксировать проектные решения, но не подтверждает ФСНБ/ГЭСН/КСИ как официальный источник.

LLM-generated content запрещен как evidence.

## 3. Статусы

- `inbox` — файл или запись поступили, но не проверены;
- `draft` — структура записи подготовлена без доверенного статуса;
- `under_review` — источник ожидает пользовательской проверки;
- `accepted` — источник принят пользователем как применимый источник учета;
- `rejected` — источник отклонен;
- `forbidden` — источник запрещен для evidence;
- `requires_norm_review` — требуется отдельная нормативная проверка.

## 4. Checksum и версия

Каждый источник перед приемкой должен иметь `version_label`, `file_path`, `checksum_sha256` и `checksum_status`.

`accepted` status без checksum запрещен.

## 5. Запреты

Запрещено:

- придумывать КСИ-коды, ГЭСН-коды или ФСНБ-версии;
- считать `llm_generated` официальным источником;
- считать `user_decision` официальным нормативным источником;
- считать `project_dictionary` официальным нормативным источником;
- создавать active matching rules из reference-intake manifest;
- использовать customer-specific data в manifest.
