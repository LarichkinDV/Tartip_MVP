# Outline ежемесячной защиты: инфраструктурный контур

## Слайд 1. Задача месяца

- Подготовить инфраструктурный контур прикладного прототипа ТАРТИП.
- Зафиксировать локальное развертывание, проверяемость и трассируемость.
- Подготовить основу для решения о дальнейшей разработке.

## Слайд 2. Что создано

- Репозиторий и базовый контур разработки.
- Governance files: project state, execution packets, artifact registry, traceability matrix.
- Acceptance, verification, user action и user review dashboards.
- Policy notes по legal/data boundary.
- Месячные документы: БФТ, ТЗ, протокол испытаний.

## Слайд 3. Почему это снижает риск разработки прототипа

- Есть source-of-truth для состояния проекта.
- Есть проверяемые команды `make verify` и `make check`.
- User-owned поля отделены от действий Codex.
- Accepted artifacts защищены от незаметного изменения.
- Реальные данные Заказчика не попадают в репозиторий.

## Слайд 4. Какие документы подготовлены

- БФТ инфраструктурного контура.
- ТЗ инфраструктурного контура.
- Протокол испытаний инфраструктурного контура.
- Outline для ежемесячной защиты.

## Слайд 5. Как проверяется результат

- Проверка структуры документов.
- Проверка отсутствия запрещенной customer-facing рамки.
- Проверка `make verify`.
- Проверка `make check`.
- Ручная приемка пользователем.

## Слайд 6. Какие ограничения сохраняются

- Нет промышленной версии программного продукта.
- Нет импорта реальных КСИ/ФСНБ/ГЭСН источников.
- Нет active matching rules.
- Нет использования реальных данных работодателя или Заказчика.
- Нет внешней передачи данных без отдельного правового основания.
